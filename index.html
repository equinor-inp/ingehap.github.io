4.7 CS - Testing

4.7.1 CS - Testing - Terminology

  
  fixture                   setup/teardown functions or representing data/resources
  functional test           test a single bit of functionality
  hook functions            way of inserting code in pytest executions
  integration test          testing a large subsystem, several unit tests
  session                   one invocation of pytest
  subcataneous test         test that does not run GUI, but below end-user interface
  system test (end-to-end)  testing the complete system 
  unit test                 testing fcunction / module / class in isolation

4.7.3 CS - Testing - Python Testing Library pytest

  • pytest - homepage
  
  How to run pytest
  -----------------
  pytest                            looks in current folder and in subfolders for test files
                                    runs the test code it finds
  pytest --help                     prints help information
  pytest "directories"              as pytest, but with roots as given
  pytest "file names sep by space"  run all tests in the files
  pytest -v test_a.py::test_b       run function test_b located in file test_a
  pytest --version                  pytest lib version
  pytest --tb=no                    turns off stak trace
  pytest -q                         decrease verbosity
  pytest -l                         show locals in tracebacks
  pytest -x                         stops if a test fails
  pytest --collect-only             only collect tests, don't execute
  
  Naming conventions
  ------------------
  Default rules for making code discoverable by pytest are
  - name test files test_*.py or *_test.py
  - name test functions / modules test_*
  - name test classes Test*
  The default rules may be altered via a onfiguration file
  
  Understanding pytest output
  ---------------------------
  F = failures
  E = errors
  s = skips (defined by decorators @pytest.mark.skip() or @pytest.mark.skipif())
  x = xfails (expected to fail and fails; defined by @pytest.mark.xfail())
  X = xpasses (expected to fail, but passes)
  
  
  Configuration files
  -------------------
  pytest.ini        directives for changing behavior of pytest
  conftest.py       hook functions and fixtures (one in test, one in subdirs)
  toc.ini
  setup.cfg
  
  
  Compatibility with other test libraries
  ---------------------------------------
  pytest can run tests written for unittest and nose packages
  
  Marking test functions by decorators
  ------------------------------------
  @pytest.mark.simple
  def test_anything():
  ..
  >> pytest -v -m 'simple'
  
  Skipping test functions by decorators
  -------------------------------------
  @pytest.mark.skip(reason='bad code, must be fixed')
  def test_meaning_of_life():
  ..
  @pytest.mark.skipif(tasks.__version__ < '0.2.0')
  def test_other():
  >> pytest -rs
  
  Marking tests as expecting to fail
  ----------------------------------
  @pytest.mark.xfail(reason='missing input')
  def test_one():

4.9.2 CS - Version Control - Git

  
  Download
  --------
  https://git-scm.com/downloads

  Set-up
  ------
  git config --global user.name "Inge H. A. Pettersen"
  git config --global user.email ingehap@gmail.com
  
  Check set-up
  ------------
  git config user.name
  git config user.email
  
  Clone a repo
  ------------
  git clone "clone url"
  
  Check out code at a particular stage
  ------------------------------------
  git checkout "commit hash"
  
  Open pull request
  -----------------
  git remote set-url origin "forked url"
  git checkout -b test-branch-2
  git commit --allow-empty -m "opening pr"
  git push origin test-branch-2
  On GitHub, click "Compare & pull request" on test-branch-2.
  Afterwards change base repository to forkked one.

4.10 CS - Virtual environments

4.10.1 CS - Virtual environments - conda

  
  Create virtual environment (default method)
  -------------------------------------------
  conda create --name "your env name" python=3.7 -y
  
  Create virtual environment from YAML file
  -----------------------------------------
  conda env create -f environment.yml
  
  Activate virtual environment
  ----------------------------
  conda activate "your env name"
  
  Deactivate virtual environment
  ------------------------------
  conda deactivate
  
  Information about current virtual environment
  ---------------------------------------------
  conda info --env
  
  Duplicate current env to YAML file
  ----------------------------------
  conda env export > my_environment.yml
  
  Remove environment
  ------------------
  conda remove --name "your env name" --all

4.10.2 CS - Virtual environments - venv

  
  Create environment
  ------------------
  python -m venv demo_env
  
  Activate environment
  --------------------
  demo_env\Scripts\activate
  
  Deactivate
  ----------
  deactivate
  
  Make requirement file
  ---------------------
  pip freeze > requirements.txt
  
  Run requirement file
  --------------------
  pip install -r requirements.txt
  
  Remove virtual environment
  --------------------------
  delete folder containing the environment
